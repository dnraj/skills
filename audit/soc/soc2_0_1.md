# SOC 2 Managed Services Compliance

## When to Use This Skill

Use this skill when the user asks for SOC 2 compliance support for a managed services organization, including:

- Building SOC 2 from the ground up for a managed services provider, MSP, MSSP, cloud operations provider, IT outsourcing provider, or help desk organization.
- Creating a SOC 2 Type I or Type II implementation roadmap.
- Building or reviewing a SOC 2 control matrix for managed services.
- Preparing evidence request lists, audit-readiness plans, remediation trackers, or control narratives.
- Mapping managed services operations to SOC 2 domains such as logical access, change management, incident response, vulnerability management, vendor management, HR security, data protection, customer environment access, and service availability.
- Helping leadership understand how SOC 2 should operate as an ongoing compliance program, not only as an audit event.

Do not present the output as an official audit opinion, legal advice, or CPA attestation guidance. For final audit conclusions, advise the user to confirm scope, criteria, evidence expectations, and report treatment with their CPA firm or qualified SOC 2 auditor.

## Core Principles

1. Treat SOC 2 as an operating program for security, reliability, confidentiality, and customer trust.
2. Start with scope before designing controls.
3. Make every control testable by defining owner, frequency, population, evidence, system, and exception handling.
4. Favor sustainable controls embedded in normal managed services workflows, especially ticketing, identity management, monitoring, change management, and vendor review processes.
5. Distinguish Type I readiness from Type II operating effectiveness.
6. Call out assumptions, gaps, dependencies, and auditor validation points.

## Initial Scoping Questions

Ask only when the missing information materially changes the output. If the user wants a quick draft, make reasonable assumptions and label them.

Key questions:

- Is the target report SOC 2 Type I, Type II, or both?
- What is the target audit period?
- Which Trust Services Criteria are in scope: Security, Availability, Confidentiality, Processing Integrity, Privacy?
- Which managed services are in scope: help desk, infrastructure management, cloud operations, endpoint management, backup, monitoring, SOC/MSSP, security operations, or customer administration?
- Which systems are in scope: identity provider, ticketing, RMM, EDR, SIEM, cloud platforms, password vault, backup tools, monitoring, documentation, HRIS, finance/procurement, source control, CI/CD, or customer portals?
- Does the organization access customer production environments or customer confidential data?
- Are contractors, subcontractors, or offshore teams involved?
- Does the company have existing policies, tickets, logs, access reviews, vendor reviews, or audit evidence?

## Managed Services Scope Model

Use this scope model unless the user provides a different one.

### Services in Scope

- Managed IT services.
- Help desk and support.
- Remote monitoring and management.
- Cloud infrastructure administration.
- Backup and recovery services.
- Endpoint management.
- Security monitoring or MSSP services, if applicable.
- Customer environment administration.
- Customer-impacting change execution.

### Systems in Scope

- Identity provider and SSO.
- Ticketing and service management platform.
- RMM or remote support tools.
- Endpoint management and EDR tools.
- Monitoring and alerting tools.
- SIEM or logging platform, if applicable.
- Cloud consoles used for customer or internal services.
- Password vault or privileged access management platform.
- Backup and recovery tools.
- Documentation and knowledge management platforms.
- HRIS and training platforms.
- Vendor management records.

### People in Scope

- Employees and contractors with access to customer systems or confidential data.
- Service desk staff.
- System administrators.
- Cloud administrators.
- Security operations personnel.
- Managers who approve access, changes, incidents, or vendors.
- Compliance, HR, procurement, and leadership roles involved in control operation.

## SOC 2 Domains for Managed Services

Organize work using these control domains.

### Governance and Risk Management

Expected controls:

- Security governance ownership is assigned.
- Policies are approved, maintained, and reviewed.
- Risk assessments are performed and tracked.
- Management reviews security, compliance, and control status.
- SOC 2 readiness and remediation activities are tracked.

Evidence:

- Approved policies.
- Risk register.
- Management meeting minutes.
- Control matrix.
- Remediation tracker.
- Security committee notes.

### Logical Access

Expected controls:

- Access requests are approved before provisioning.
- MFA is enforced for in-scope systems.
- Privileged access is restricted based on job responsibilities.
- User access is reviewed periodically.
- Terminated users are removed within a defined SLA.
- Customer environment access is approved, logged, and periodically reviewed.

Evidence:

- Access request tickets.
- Approval records.
- User listings.
- MFA configuration.
- Access review exports and sign-offs.
- Termination tickets.
- Customer access records.

### Change Management

Expected controls:

- Internal and customer-impacting changes are documented.
- Changes are approved before implementation.
- Testing or validation is documented.
- Emergency changes are tracked and retrospectively reviewed.
- Customer approvals are retained when required.

Evidence:

- Change tickets.
- Pull requests.
- Deployment logs.
- Testing results.
- Customer approval tickets.
- Emergency change review records.

### System Operations and Monitoring

Expected controls:

- Critical systems and customer-impacting services are monitored.
- Alerts are triaged and tracked.
- Incidents are documented, escalated, resolved, and reviewed.
- Backups are monitored and restore tests are performed.
- Capacity, availability, and service health are reviewed.

Evidence:

- Monitoring dashboards.
- Alert logs.
- Incident tickets.
- Post-incident reviews.
- Backup job logs.
- Restore test evidence.
- Uptime or availability reports.

### Vulnerability and Security Operations

Expected controls:

- Vulnerability scans are performed on in-scope assets.
- Findings are prioritized by severity.
- Remediation is tracked to closure.
- Security tools are monitored.
- Exceptions are risk-accepted and reviewed.

Evidence:

- Scan reports.
- Remediation tickets.
- Patch reports.
- Exception approvals.
- EDR or SIEM alerts.
- Security review records.

### Vendor and Subcontractor Management

Expected controls:

- Vendors are inventoried and risk-ranked.
- Critical vendors are reviewed before onboarding and periodically thereafter.
- Vendor security documentation is retained.
- Subcontractors are subject to confidentiality, access, training, and offboarding controls.
- Subprocessors are tracked when applicable.

Evidence:

- Vendor inventory.
- Risk assessments.
- SOC reports, ISO certificates, questionnaires, or security documentation.
- Contracts, DPAs, or confidentiality agreements.
- Annual vendor review sign-offs.

### HR and Workforce Security

Expected controls:

- Background checks are performed where lawful and applicable.
- Employees and contractors sign confidentiality or acceptable use agreements.
- Security awareness training is completed upon hire and annually.
- Onboarding and offboarding workflows are documented.
- Role changes trigger access updates.

Evidence:

- Training reports.
- Signed agreements.
- Background check completion records.
- HRIS records.
- Onboarding and offboarding tickets.

### Data Protection and Confidentiality

Expected controls:

- Customer data and internal data are classified.
- Encryption is used in transit and at rest where applicable.
- Customer credentials are stored only in approved secure systems.
- Data retention and deletion processes are defined.
- Customer data access and handling expectations are documented.

Evidence:

- Data classification policy.
- Encryption configuration.
- Password vault records.
- Retention schedule.
- Data deletion tickets.
- Customer data handling procedures.

### Business Continuity and Availability

Expected controls:

- Business continuity and disaster recovery plans are documented.
- Backup and recovery responsibilities are defined.
- Restore tests are performed.
- Availability commitments are monitored.
- Service incidents are reviewed for customer impact.

Evidence:

- BCP/DR plan.
- Backup logs.
- Restore test records.
- Uptime reports.
- Incident postmortems.
- Service review meeting notes.

## Ground-Up Implementation Roadmap

Use this default roadmap for a managed services organization starting from zero.

### Phase 1: Scope and Governance

Activities:

- Define SOC 2 report type, criteria, audit period, in-scope services, systems, people, vendors, and locations.
- Assign executive sponsor, SOC 2 program owner, and control owners.
- Create a project plan, evidence calendar, and remediation tracker.
- Select or engage a CPA firm or readiness advisor.

Deliverables:

- SOC 2 scope statement.
- In-scope system inventory.
- Control owner roster.
- Initial project plan.
- Auditor scoping notes.

### Phase 2: Readiness Gap Assessment

Activities:

- Review current policies, tickets, systems, and evidence.
- Assess access, change, incident, vulnerability, backup, vendor, HR, and data protection processes.
- Identify design gaps and evidence gaps.
- Prioritize remediation based on audit risk and operational risk.

Deliverables:

- Gap assessment.
- Remediation roadmap.
- Evidence readiness checklist.
- Updated control matrix.

### Phase 3: Policy and Control Design

Activities:

- Draft or update core policies.
- Define control activities with owner, frequency, evidence, system, and exception handling.
- Align controls to actual managed services workflows.
- Build evidence templates for recurring controls.

Deliverables:

- Approved policy set.
- SOC 2 control matrix.
- Evidence request list.
- Control narratives.

### Phase 4: Control Implementation

Activities:

- Implement MFA, access request workflows, access reviews, privileged access restrictions, and termination procedures.
- Implement change management, customer approval records, emergency change procedures, and deployment evidence.
- Implement incident, vulnerability, backup, restore testing, monitoring, and vendor review processes.
- Train control owners on evidence requirements.

Deliverables:

- Operating controls.
- Completed first-cycle access review.
- Change and incident workflows.
- Vendor risk reviews.
- Training completion records.

### Phase 5: Type I Readiness

Activities:

- Confirm controls are designed and implemented.
- Collect point-in-time evidence.
- Resolve major design gaps.
- Prepare auditor walkthrough materials.

Deliverables:

- Type I evidence package.
- Final control matrix.
- Management walkthrough notes.
- Open item tracker.

### Phase 6: Type II Operating Period

Activities:

- Operate controls consistently across the observation period.
- Collect monthly, quarterly, annual, and per-occurrence evidence.
- Track exceptions and remediation.
- Perform internal compliance check-ins.

Deliverables:

- Type II evidence repository.
- Completed evidence calendar.
- Exception log.
- Remediation closure records.

### Phase 7: Audit Fieldwork and Reporting

Activities:

- Provide evidence samples.
- Respond to auditor requests.
- Explain exceptions and remediation.
- Review draft report language with management and auditor.

Deliverables:

- Auditor request responses.
- Sample evidence packages.
- Management representation support.
- Final SOC 2 report.

| Control ID | Domain | Control Objective | Control Activity | Owner | Frequency | Evidence | Systems | Design Notes | Operating Effectiveness Notes |
|---|---|---|---|---|---|---|---|---|---|
| SOC2-GOV-01 | Governance | Establish accountability for security and compliance | Management assigns a SOC 2 program owner and control owners for in-scope domains | Executive Sponsor / Compliance | Annual and upon changes | Control owner roster, org chart, meeting notes | Governance records | Confirm named accountable roles | Test whether ownership was maintained during the period |
| SOC2-ACC-01 | Logical Access | Ensure only authorized personnel receive access | Access requests are approved by the appropriate manager or system owner before provisioning | IT / System Owner | Per request | Access ticket, approval, user listing | IdP, ticketing | Approval must occur before provisioning | Sample new users and verify approval timing |
| SOC2-CHG-01 | Change Management | Ensure customer-impacting changes are authorized | Customer-impacting changes are documented, approved, tested or validated, and retained in the ticketing system | Operations / Service Manager | Per change | Change ticket, customer approval, test evidence | Ticketing, cloud tools | Define what qualifies as customer-impacting | Sample changes from complete population |

## Common Managed Services Red Flags

Flag these issues when present:

- Remote support access exists but is not periodically reviewed.
- Customer credentials are stored in spreadsheets, email, chat, or personal notes.
- Customer-impacting changes are performed without documented approval.
- Help desk tickets are closed without evidence of approval, validation, or customer communication.
- Terminated users are removed from internal tools but not from customer environments.
- Contractors have access but are not included in training, access reviews, confidentiality agreements, or offboarding.
- Monitoring alerts exist but triage evidence is not retained.
- Backups run but restore tests are not performed.
- Vendor inventory excludes critical SaaS, RMM, EDR, cloud, or subcontractor providers.
- Access reviews are based on screenshots instead of complete user populations.
- Vulnerability scans exist but remediation is not tracked to closure.

## Output Guidance

For user-facing deliverables:

- Use professional, audit-ready language.
- Use tables for roadmaps, control matrices, evidence lists, remediation trackers, and gap assessments.
- Include assumptions when facts are missing.
- Use placeholders such as `TBD`, `[Owner]`, `[System]`, and `[Audit Period]` rather than fabricating details.
- Distinguish between design readiness for Type I and operating effectiveness for Type II.
- Recommend auditor validation for final scope and audit treatment.
