# SOC 2 Implementation Roadmap and Control Matrix for a Managed Services Company

## Assumptions

This roadmap assumes the organization is a managed services provider that supports customer IT environments, cloud infrastructure, endpoints, monitoring, backup, help desk, and administrative support. It assumes the organization is building SOC 2 from the ground up and is likely targeting Security as the baseline Trust Services Criteria, with Availability and Confidentiality as likely additional criteria depending on customer commitments and data handling.

This document is designed for readiness and implementation planning. Final SOC 2 scope, control descriptions, evidence expectations, and audit treatment should be validated with the CPA firm performing the SOC 2 examination.

## Target SOC 2 Scope

| Scope Area | Recommended Inclusion | Notes |
|---|---|---|
| Report Type | Type I first, then Type II | Type I validates design at a point in time; Type II validates operating effectiveness over an observation period. |
| Trust Services Criteria | Security, Availability, Confidentiality | Security is required. Availability is recommended if uptime or service commitments are material. Confidentiality is recommended if customer confidential data or credentials are handled. |
| Services | Managed IT, cloud administration, help desk, monitoring, endpoint management, backup, customer environment support | Exclude services only if they are clearly segregated and not relevant to customer commitments. |
| Systems | IdP/SSO, ticketing, RMM, EDR, monitoring, SIEM/logging, password vault, backup platform, cloud consoles, documentation platform, HRIS, training platform | Maintain a formal in-scope system inventory. |
| People | Employees, contractors, administrators, service desk, security operations, managers, compliance, HR, procurement | Include contractors and subcontractors if they access customer systems or data. |
| Vendors | Cloud providers, RMM, EDR, ticketing, backup, monitoring, password vault, HRIS, training, subcontractors | Critical vendors should be risk-ranked and reviewed. |
| Customer Environments | Included where the company administers or accesses customer systems | Customer access is a managed-services-specific SOC 2 focus area. |

## Implementation Roadmap

| Phase | Timeline | Objectives | Key Activities | Deliverables | Primary Owners |
|---|---:|---|---|---|---|
| Scope and Governance | Weeks 1-2 | Define the SOC 2 program boundaries and accountability model | Select report type, criteria, services, systems, vendors, locations, and people in scope; assign executive sponsor, SOC 2 owner, and control owners; choose audit firm or readiness advisor | Scope statement, system inventory, vendor inventory, control owner roster, project plan | Executive Sponsor, Compliance, Security, IT |
| Readiness Gap Assessment | Weeks 2-4 | Identify design and evidence gaps before the audit period | Review policies, tickets, access workflows, change records, incident processes, vulnerability practices, vendor reviews, HR processes, and backup evidence | Gap assessment, remediation tracker, risk-prioritized action plan | Compliance, Security, IT, Operations |
| Policy Foundation | Weeks 3-6 | Create the required governance documentation | Draft, approve, and publish policies for information security, access control, acceptable use, change management, incident response, vulnerability management, vendor management, risk management, data classification, backup, BCP/DR, and HR security | Approved policy set, policy review schedule, employee acknowledgment evidence | Compliance, Security, HR, Leadership |
| Control Design | Weeks 4-7 | Convert policy expectations into testable controls | Define control objective, activity, owner, frequency, system, evidence, population, and exception handling for each control | Control matrix, evidence request list, control narratives | Compliance, Security, IT, Operations |
| Access Control Implementation | Weeks 5-10 | Build reliable identity and access controls | Enforce MFA; formalize access request approvals; restrict privileged access; implement quarterly access reviews; define termination SLA; review customer environment access | MFA evidence, access request workflow, privileged user list, first access review, termination checklist | IT, Security, HR, System Owners |
| Change Management Implementation | Weeks 6-10 | Formalize internal and customer-impacting change control | Require change tickets, approvals, validation, deployment records, emergency change review, and customer approval where applicable | Change workflow, emergency change procedure, change population report, sample change evidence | Operations, IT, Engineering, Service Managers |
| Security Operations Implementation | Weeks 6-12 | Operationalize monitoring, vulnerabilities, incidents, and backups | Implement alert triage, incident ticketing, vulnerability scanning, remediation SLAs, backup monitoring, restore testing, and service availability reporting | Incident log, alert records, vulnerability reports, remediation tickets, backup logs, restore test evidence | Security, IT, Operations |
| Vendor and Contractor Controls | Weeks 7-12 | Control third-party and subcontractor risk | Build vendor inventory, assign risk ratings, collect assurance documentation, review critical vendors, confirm contractor onboarding/offboarding requirements | Vendor inventory, vendor risk reviews, SOC reports/certifications, contractor access and training records | Procurement, Compliance, Legal, Security |
| Training and HR Controls | Weeks 7-12 | Ensure workforce security expectations are evidenced | Launch security awareness training; retain confidentiality agreements; document onboarding and offboarding; reconcile terminations to access removal | Training reports, signed agreements, onboarding records, termination evidence | HR, IT, Security |
| Type I Readiness | Weeks 12-16 | Confirm controls are designed and implemented | Collect point-in-time evidence; perform management walkthroughs; resolve high-priority gaps; prepare auditor package | Type I evidence package, updated control matrix, walkthrough notes, open item tracker | Compliance, Control Owners, Auditor |
| Type II Observation Period | Months 4-9 or 4-15 | Operate controls consistently over time | Run monthly, quarterly, annual, and per-occurrence controls; collect evidence; track exceptions; remediate issues | Evidence repository, completed access reviews, change samples, incident records, vendor reviews, exception log | Control Owners, Compliance |
| Audit Fieldwork and Report | End of Observation Period | Complete audit testing and reporting | Respond to evidence requests; provide samples; explain exceptions; review draft report language | Auditor response tracker, final evidence package, management responses, SOC 2 report | Compliance, Leadership, Auditor |

## First 30/60/90-Day Action Plan

| Period | Priority Actions | Success Criteria |
|---|---|---|
| First 30 Days | Confirm SOC 2 criteria and scope; assign executive sponsor and SOC 2 owner; create inventories for systems, vendors, employees, contractors, and customer environment access; select audit firm or readiness advisor; start gap assessment | Scope is documented, owners are assigned, inventories exist, and the gap assessment has begun |
| Days 31-60 | Approve core policies; design control matrix; implement ticket-based access requests, change requests, incident tracking, and vendor reviews; enforce MFA for in-scope systems; define evidence calendar | Policies are approved, controls are documented, MFA is enforced, and recurring evidence expectations are scheduled |
| Days 61-90 | Complete first access review; complete first vendor review cycle for critical vendors; perform vulnerability scan and remediation tracking; perform backup restore test; complete security awareness training; prepare Type I readiness package | First-cycle evidence exists across access, vendor, vulnerability, backup, HR, and governance controls |

## SOC 2 Control Matrix

| Control ID | Domain | Control Objective | Control Activity | Owner | Frequency | Evidence | Systems | Design Notes | Operating Effectiveness Notes |
|---|---|---|---|---|---|---|---|---|---|
| SOC2-GOV-01 | Governance | Establish accountability for the SOC 2 program | Management assigns an executive sponsor, SOC 2 program owner, and control owners for in-scope domains | Executive Sponsor / Compliance | Annual and upon material changes | Control owner roster, org chart, meeting notes | Governance records | Owners should be named roles with clear responsibilities | Test whether ownership was maintained and updated during the period |
| SOC2-GOV-02 | Governance | Ensure policies are approved and current | Security and operational policies are reviewed, approved by management, and updated at least annually | Compliance / Security | Annual | Approved policies, version history, review sign-off | Policy repository | Policies should align with actual operations | Test annual review and approval evidence |
| SOC2-GOV-03 | Risk Management | Identify and manage security and operational risks | The organization performs a risk assessment and tracks mitigation actions to closure or risk acceptance | Security / Compliance | Annual and upon material changes | Risk register, remediation tracker, risk acceptance records | GRC tool or spreadsheet | Risks should cover managed services and customer access | Test risk review date, owner, status, and remediation |
| SOC2-GOV-04 | Management Oversight | Provide oversight over security and compliance | Management reviews security, compliance, incidents, risks, and remediation status | Leadership / Compliance | Quarterly | Meeting agenda, minutes, metrics, action items | Governance records | Include SOC 2 status and unresolved exceptions | Test quarterly meeting evidence and follow-up actions |
| SOC2-ACC-01 | Logical Access | Ensure only authorized users receive access | Access requests are documented and approved by the appropriate manager or system owner before provisioning | IT / System Owner | Per request | Access ticket, approval record, provisioning evidence | IdP, ticketing, in-scope systems | Approval must occur before access is granted | Sample new users and verify approval timing |
| SOC2-ACC-02 | Logical Access | Reduce unauthorized access risk | MFA is enforced for in-scope systems, including privileged access and remote access tools | IT / Security | Continuous | MFA configuration export, conditional access settings | IdP, VPN, RMM, cloud consoles | Exceptions should be approved and time-bound | Test MFA settings and exception records |
| SOC2-ACC-03 | Privileged Access | Restrict administrative access | Privileged access is limited to authorized personnel based on job responsibilities and approved business need | Security / IT | Per request and quarterly review | Privileged user listing, access approvals, review sign-off | IdP, cloud consoles, RMM, password vault | Define privileged roles clearly | Sample privileged users and verify approval and business need |
| SOC2-ACC-04 | Access Reviews | Detect inappropriate access | System owners review access for in-scope systems and customer environment access on a defined cadence | System Owners / Security | Quarterly | User listing, review sign-off, exception log, remediation tickets | IdP, ticketing, RMM, customer consoles | Use complete user populations, not screenshots only | Test review completion, reviewer, population, and exception closure |
| SOC2-ACC-05 | Terminations | Remove access promptly when personnel leave | HR notifies IT of terminations, and IT removes access from in-scope internal and customer systems within the defined SLA | HR / IT | Per termination | HR termination record, offboarding ticket, deactivation evidence | HRIS, IdP, ticketing, customer systems | Include contractors and customer environment access | Sample terminations and verify timely removal |
| SOC2-ACC-06 | Customer Environment Access | Control access to customer systems | Access to customer environments is approved, limited to authorized personnel, logged where available, and periodically reviewed | Operations / Security | Per request and quarterly review | Customer access approval, user listing, logs, review sign-off | RMM, customer portals, cloud consoles | This is critical for managed services scope | Test access approvals and periodic reviews |
| SOC2-CHG-01 | Change Management | Ensure changes are authorized | Internal and customer-impacting changes are documented and approved before implementation | Operations / Service Manager | Per change | Change ticket, approval record, change population report | Ticketing, source control, cloud tools | Define standard, normal, and emergency changes | Sample changes from full population |
| SOC2-CHG-02 | Change Testing | Reduce change-related failures | Changes are tested or validated before deployment, and results are documented | IT / Operations / Engineering | Per change | Test results, validation notes, deployment checklist | Ticketing, CI/CD, cloud tools | Validation may be technical or operational depending on service | Test evidence of validation before closure |
| SOC2-CHG-03 | Customer Approval | Ensure customer authorization for customer-impacting work | Customer-impacting changes require customer approval or documented authorization before implementation | Service Manager / Account Owner | Per applicable change | Customer approval, ticket notes, statement of work reference | Ticketing, customer portal | Define what requires explicit customer approval | Sample customer-impacting changes |
| SOC2-CHG-04 | Emergency Changes | Maintain control during urgent changes | Emergency changes are documented, justified, approved retrospectively, and reviewed after implementation | Operations / Security | Per emergency change | Emergency change ticket, justification, retrospective approval | Ticketing | Retrospective review should occur within a defined timeframe | Test emergency change review timing |
| SOC2-OPS-01 | Monitoring | Detect operational or security events | Critical systems, managed services tools, and customer-impacting services are monitored, and alerts are triaged | Operations / Security | Continuous | Monitoring configuration, alert records, triage tickets | Monitoring, SIEM, ticketing | Define severity and escalation paths | Sample alerts and verify triage |
| SOC2-OPS-02 | Incident Response | Respond to security and service incidents | Incidents are logged, classified, escalated, resolved, and reviewed according to the incident response procedure | Security / Operations | Per incident | Incident tickets, severity classification, postmortem, communications | Ticketing, SIEM, monitoring | Include customer notification criteria | Sample incidents and verify lifecycle evidence |
| SOC2-OPS-03 | Backup Monitoring | Protect recoverability of systems and data | Backups for in-scope systems are scheduled, monitored, and failures are investigated | IT / Operations | Daily or per backup schedule | Backup job reports, failure tickets, resolution evidence | Backup platform, ticketing | Define systems covered and backup frequency | Test backup success and failure follow-up |
| SOC2-OPS-04 | Restore Testing | Validate recoverability | Restore tests are performed for in-scope systems and results are documented | IT / Operations | At least annual, or more frequently for critical systems | Restore test plan, test results, issues, remediation | Backup platform | Backup success alone is not sufficient | Test restore completion and remediation of failures |
| SOC2-OPS-05 | Availability Review | Monitor service commitments | Availability, uptime, major incidents, and service metrics are reviewed by management | Operations / Leadership | Monthly or quarterly | Uptime report, service review deck, incident summaries | Monitoring, ticketing | Needed if Availability is in scope | Test periodic review and follow-up actions |
| SOC2-VUL-01 | Vulnerability Management | Identify security weaknesses | Vulnerability scans are performed for in-scope systems and findings are documented | Security / IT | Monthly or continuous | Scan reports, asset scope, finding exports | Vulnerability scanner | Scope should align to in-scope assets | Test scan frequency and population |
| SOC2-VUL-02 | Vulnerability Remediation | Remediate vulnerabilities based on severity | Vulnerabilities are prioritized and remediated according to defined severity timelines or risk-accepted by management | Security / IT | Per finding | Remediation tickets, patch records, exception approvals | Scanner, ticketing, endpoint tools | Define severity SLAs in policy | Sample findings and verify closure or approval |
| SOC2-HR-01 | Workforce Screening | Reduce personnel risk | Background checks are performed before hire where lawful and applicable | HR | Per hire | Background check completion record | HRIS, background check provider | Account for jurisdictional limits | Sample hires and verify completion |
| SOC2-HR-02 | Confidentiality | Protect customer and company information | Employees and contractors sign confidentiality, acceptable use, or similar agreements | HR / Legal | Upon hire or engagement | Signed agreements | HRIS, e-signature platform | Include contractors and subcontractors | Sample workforce population |
| SOC2-HR-03 | Security Training | Promote secure behavior | Personnel complete security awareness training upon hire and annually | HR / Security | Upon hire and annual | Training completion report | Training platform | Include contractors if they access systems or data | Test completion for new and active personnel |
| SOC2-HR-04 | Role Changes | Keep access aligned to job duties | Role changes trigger review and update of access where responsibilities change | HR / IT / Manager | Per role change | HR change record, access update ticket | HRIS, ticketing, IdP | Often missed in growing companies | Sample role changes and verify access review |
| SOC2-VND-01 | Vendor Inventory | Maintain visibility into third-party risk | The organization maintains an inventory of vendors and subcontractors supporting in-scope services | Procurement / Compliance | Continuous, reviewed at least annually | Vendor inventory, owner assignments | Vendor tracker or GRC | Include SaaS, cloud, subcontractors, RMM, EDR, backup | Test completeness and annual review |
| SOC2-VND-02 | Vendor Risk Review | Assess critical vendor security | Critical vendors are risk-ranked and reviewed before onboarding and periodically thereafter | Procurement / Security / Compliance | Before onboarding and annual | Vendor risk assessment, SOC report, ISO certificate, questionnaire | Vendor tracker | Define criticality criteria | Sample critical vendors and verify review |
| SOC2-VND-03 | Subcontractor Controls | Manage outsourced service risk | Subcontractors with access to customer systems or data complete required onboarding, confidentiality, training, access approval, and offboarding controls | Operations / HR / Compliance | Per engagement | Contract, confidentiality agreement, training record, access approval | HRIS, ticketing, contract repository | Treat subcontractors like in-scope personnel | Sample subcontractors |
| SOC2-DAT-01 | Data Classification | Define data handling expectations | The organization classifies customer, internal, confidential, and restricted data and defines handling requirements | Security / Compliance | Annual review | Data classification policy, data inventory | Policy repository | Align with customer data handled in managed services | Test policy review and data inventory |
| SOC2-DAT-02 | Encryption | Protect data in transit and at rest | In-scope systems encrypt customer and confidential data in transit and at rest where applicable | IT / Security | Continuous | Encryption configuration, TLS settings, cloud encryption settings | Cloud platforms, SaaS tools | Document exceptions and compensating controls | Test configuration evidence |
| SOC2-DAT-03 | Credential Management | Protect customer credentials | Customer credentials are stored only in approved password vault or privileged access management tools with restricted access | IT / Security / Operations | Continuous and quarterly review | Vault configuration, access listing, access review evidence | Password vault, PAM | Do not store credentials in spreadsheets or chat | Test vault access and review records |
| SOC2-DAT-04 | Retention and Deletion | Limit unnecessary data retention | Customer and company data are retained and deleted according to documented retention requirements | Compliance / IT / Operations | Per request and scheduled review | Retention policy, deletion tickets, disposal records | Storage, ticketing, documentation tools | Define customer offboarding handling | Sample deletion or offboarding records |
| SOC2-BCP-01 | Business Continuity | Prepare for service disruption | The organization maintains a business continuity and disaster recovery plan for in-scope services | Leadership / IT / Operations | Annual review | BCP/DR plan, approval record, review history | Policy repository | Include roles, communication, recovery priorities | Test annual review evidence |
| SOC2-BCP-02 | BCP/DR Testing | Validate continuity procedures | Business continuity or disaster recovery procedures are tested or tabletoped, and findings are tracked | IT / Operations / Leadership | Annual | Test plan, tabletop notes, issues, remediation tracker | Governance records | Tie results to remediation actions | Test completion and issue closure |

## Remediation Roadmap

| Priority | Workstream | Common Starting Gap | Remediation Action | Target Timing | Success Criteria |
|---|---|---|---|---|---|
| High | Scope | No formal SOC 2 boundary | Define services, systems, people, vendors, and customer access in scope | Weeks 1-2 | Written scope approved by leadership |
| High | Access | Access is granted informally | Require ticketed approval before provisioning | Weeks 4-8 | New user samples have documented approval before access |
| High | MFA | MFA not enforced everywhere | Enforce MFA for in-scope systems and privileged access | Weeks 4-8 | MFA configuration exports show enforcement |
| High | Terminations | Offboarding does not include customer systems | Add internal and customer environment access removal checklist | Weeks 5-9 | Termination samples show timely access removal |
| High | Customer Access | Customer environment access is not reviewed | Create quarterly customer access review process | Weeks 6-10 | First review completed with exceptions tracked |
| High | Change | Customer-impacting changes lack approval evidence | Require change tickets and customer authorization where applicable | Weeks 6-10 | Change samples include approval and validation |
| High | Vulnerability | Scans exist but remediation is not tracked | Create severity-based remediation workflow | Weeks 6-12 | Findings are tracked to closure or risk acceptance |
| Medium | Vendor | Vendor list is incomplete | Create risk-ranked vendor inventory and critical vendor review workflow | Weeks 7-12 | Critical vendors have review evidence |
| Medium | Backup | No restore test evidence | Perform and document restore testing | Weeks 8-12 | Restore test record includes results and issues |
| Medium | Training | Training is ad hoc | Launch required security awareness training | Weeks 7-12 | Completion report covers active personnel |
| Medium | BCP/DR | Plan is missing or untested | Create BCP/DR plan and run tabletop | Weeks 10-16 | Test notes and remediation actions exist |

## Auditor Readiness Notes

- Every control should have a clear owner, frequency, population, evidence source, and exception process.
- Use complete exports for populations, such as all users, all changes, all incidents, all vendors, or all terminations.
- Retain evidence as it is produced. Do not recreate evidence after the fact.
- Make sure policy requirements match actual practice. Auditors will test what the organization says it does.
- Track exceptions transparently with risk assessment, remediation owner, target date, and closure evidence.
- Validate final Trust Services Criteria, report period, control wording, and evidence expectations with the CPA firm before audit fieldwork.
